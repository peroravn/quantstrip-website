STRATEGY AUTOMATION

Continuous execution via persistent background services

Each strategy is deployed as an isolated process — referred to as a client — registered with a system-level background service that initialises on startup and maintains execution across sessions. Clients are fully isolated: each has its own schedule, log stream, and database connection with no shared state between processes.

The Client Monitor provides a consolidated operational view: active clients, health status, start timestamps, and runtime error reporting — accessible via a locally hosted browser interface. Client lifecycle operations require no terminal access.

DASHBOARD APPS

Integrated operational interface

The Quantstrip dashboard is a locally hosted web application providing a suite of purpose-built tooling: a Python IDE for authoring and deploying client code, a SQL editor for direct database access, a portfolio reporting module, a notification centre, and a settings manager.

All applications read from and write to the same underlying data model. Query results in the SQL editor reflect the same records rendered in the reporting module — there is no data transformation layer or synchronisation step between views.

DATA MODEL

Canonical trade lifecycle schema

Orders, executions, commissions, and position events are persisted to a structured SQLite database that is bundled with the platform. The schema is normalised around the complete trade lifecycle — from order submission through execution confirmation to commission settlement — providing a consistent, audit-ready record without requiring user-defined table structures.

PYTHON API

A focused library with a narrow, well-defined scope

The Quantstrip Python library exposes five modules covering all platform integration points. The interface is intentionally narrow: rather than a general-purpose framework, it provides structured access to scheduling, database operations, notifications, configuration, and logging — the operational concerns common to all production strategies.

my_strategy.py 
from quantstrip import ClientBase, db_handler
from quantstrip import email_manager, settings
import logging

# Every strategy is a class that extends ClientBase.
# The base class wires up scheduling, logging,
# database access, and graceful shutdown for you.

class MyStrategy(ClientBase):

    def __init__(self, *args):
        super().__init__()
        self.display_name = "My Strategy"
        self.db = db_handler
        self.scheduler.every(1).day.at("16:00").do(self.job)

    def job(self):

                signal = self.compute_signal()
                self.db.insert_strategy_event(
            strategy_id="my_strat",
            symbol="SPY",
            event_type=signal
                )
                email_manager.send_summary(signal)

Five modules. Everything handled.

ClientBase
Base class for all connected clients. Provides built-in scheduling, lifecycle hooks, error handling, and integration with the background service.
db_handler
CRUD methods for every table in the canonical data model — orders, executions, positions, strategy events, commissions, and more.
email_manager
Compose and send structured email summaries and alerts. Supports multiple SMTP accounts and configurable recipients per strategy.
settings
Read and write from the platform settings tree at runtime — credentials, parameters, flags — without touching config files directly.
logger
Central rotating log system. Every client writes to its own log stream, visible live in the dashboard and preserved for post-trade review.

BROKER INTEGRATION

Interactive Brokers integration

Quantstrip includes a pre-built adapter for Interactive Brokers, implemented against the official IB API. The adapter manages connection state and reconnection, and maps IB native events — order status changes, execution reports, commission notifications, and position updates — directly to the Quantstrip canonical data model.

A reference library of IB-specific example clients covers the most common execution workflows, providing a structured starting point for integration with existing strategies.

Interactive Brokers

  • Real-time execution and commission events
  • Historical OHLCV data retrieval
  • Position and account reconciliation
  • Automatic reconnect on disconnect
POST-TRADE REPORTING

Portfolio performance reporting

The Reporting module aggregates PnL across all registered strategies and produces cumulative equity curves, drawdown analysis, and return distribution histograms over configurable time horizons. Per-strategy statistics include annualised Sharpe ratio, maximum drawdown, trade count, and win rate.

All reporting data is derived directly from the platform database. There is no intermediate data preparation step — reported figures are consistent with the raw execution records accessible via the SQL editor.
